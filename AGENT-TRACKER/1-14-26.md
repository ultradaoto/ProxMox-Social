## Proposed Architecture: Hybrid Hard-Coded + Self-Healing Vision

### Concept
1. **Primary:** Hard-coded Python coordinates for all clicks/actions
   - Fast execution (no API calls)
   - Predictable behavior
   - Zero cost per action

2. **Fallback:** Vision model "self-healer" triggered on failures
   - Detects when buttons have moved
   - Recalibrates coordinates automatically
   - Only called when something breaks (cost-effective)

### Data to Store
- Absolute pixel coordinates for each action
- Element names/labels for visual matching
- Timing delays between steps
- Historical success/failure rates per coordinate

### Self-Healing Trigger Conditions
- [ ] Consecutive failures > threshold (e.g., 3 failures)
- [ ] Expected element not found at stored coordinates
- [ ] Vision model detects UI has changed significantly
- [ ] Manual trigger for recalibration

### Self-Healing Process
1. Take screenshot of current state
2. Call Qwen 3 Vision (expensive but accurate)
3. Ask: "Where is the [button name] on this screen? Provide exact coordinates."
4. Compare to stored coordinates
5. If delta > threshold ‚Üí update stored coordinates
6. Log the change for review
7. Resume with corrected coordinates

## Self-Healing Coordinate System - IMPLEMENTED (01/14 Afternoon Session)

### ‚úÖ What Was Built

A complete **hybrid hard-coded + self-healing vision system** for Instagram automation that:
- Stores successful click coordinates in JSON
- Uses stored coordinates as primary (fast, free, accurate)
- Falls back to vision on single failure
- Triggers self-healing after 3 consecutive failures
- Auto-recalibrates coordinates when Instagram UI changes

### Files Created

#### Core Modules (NEW)
1. **`Ubu-Cont/src/subsystems/coordinate_store.py`** (425 lines)
   - JSON-based coordinate storage with success/failure tracking
   - Thread-safe operations with automatic backups
   - Tracks consecutive failures (triggers healing at 3)
   - Healing history with before/after coordinates
   - Statistics: success rate, total clicks, healing events

2. **`Ubu-Cont/src/subsystems/self_healer.py`** (264 lines)
   - Automatic coordinate recalibration using vision API
   - Validates new coordinates (screen bounds, X range, delta)
   - Saves healing screenshots for debugging
   - Only calls expensive vision API when healing triggered

#### Integration (MODIFIED)
3. **`Ubu-Cont/src/workflows/instagram_workflow.py`**

   - **Updated 8 workflow steps** to use healing-aware clicks:

### How It Works

#### Normal Operation (Fast Path - 90% of clicks)
1. Workflow calls `_find_and_click_with_healing("click_create_post", ...)`
2. CoordinateStore returns stored coordinates (e.g., 125, 450)
3. Clicks at stored location instantly (no vision API call)
4. Records success ‚úÖ (resets failure counter)

#### When UI Drifts (Healing Path - ~10% of clicks)
1. Stored coordinates fail 3 times consecutively
2. SelfHealer triggers:
   - Takes screenshot
   - Calls vision API (Qwen 2.5-VL)
   - Gets new coordinates (e.g., 130, 455)
   - Validates: within screen bounds, within X range, delta < 200px
   - Updates CoordinateStore with healing event
3. Next click uses new coordinates
4. Healing event logged in `instagram.json` with before/after

### Expected Benefits

Based on failure analysis:

| Metric | Before | After (Estimated) |
|--------|--------|-------------------|
| Vision 2px off failures | 27% | ~0% |
| Element moved failures | 45% | ~5% |
| Vision API calls per workflow | 21 steps | ~2 steps (90% reduction) |
| **Overall failure rate** | **~72%** | **~25%** |
| **Vision API cost** | Baseline | **80-90% reduction** |
